== Initial Note ==
date: 2026-03-23

The strongest thing here is the explicit coordination layer. Most UI frameworks handle gesture conflicts through event consumption and propagation order — effectively a hidden priority system that's hard to reason about or debug. Here the judge makes conflict resolution a named, inspectable process. You can see why a gesture was denied, not just that it was. The judge-notes field extending that to runtime visibility is a natural consequence of taking the idea seriously.
The tokenizer/organism split is the other genuinely good idea. Deriving shared perceptual facts once per cycle — pressed, released, threshold-crossed, entered-target — means all organisms see a consistent world. Without this, you get subtle bugs where organism A's hit test and organism B's hit test diverge because they run at slightly different moments, or because one of them mutates state the other depends on. The tokenizers create a stable epistemic snapshot that everything downstream reasons from.
The effect routing split between world mutation and projection preview is clean. The marquee rectangle, the drag anchor line, the hover highlight — none of these need to exist in durable state, and keeping them out of it means the world is always clean and the rendering is clearly a pure read. That also means reset is trivially correct.
The ARMED intermediate state across all gesture organisms is thoughtful. Most naive implementations arm and immediately start dragging, which means accidental micro-drags trigger gestures. The threshold + ARMED pattern means intent has to be declared before resources are claimed, which reduces false positives and makes cancellation (on release while ARMED) feel natural.

Now the weaknesses:
The judge is written as a hardcoded policy function. maintain_judge and get_permission contain explicit if current == "resize-object" / if current == "drag-object" branches. This means every new organism requires surgery on the judge. The architecture implies a general coordination mechanism, but the implementation is a closed list. If you added ten more organisms, the judge would become a large brittle conditional. A better design would let organisms declare their resource requirements and conflict policies as data, with the judge evaluating them against a general rule.
The organisms communicate with the judge through mutable shared state rather than through their return value. An organism sets organism["HELD"] and then calls get_permission, which reads it back. This makes the flow implicit — the organism has to remember to populate HELD before asking, and the judge has to know which HELD fields to look at for which organism. A cleaner design would have organisms pass their resource request explicitly as an argument to get_permission, making the contract visible at the call site.
Global mutable dicts as the execution model works for a demo but creates a tension. The blackboard is conceptually the medium that organisms read and write — that's fine. But g, world, and system being module-level means every function is implicitly parameterized by all of them. Testing any organism in isolation requires constructing the entire system state. If you wanted to run two independent canvases, or test organism behavior without a Tk window, the architecture works against you. This is the primary testability cost.
The periodic tick running a full cycle unconditionally is fine here but is a latent issue. Every 100ms you run tokenizers, evaluate all organisms, and re-render, even if nothing has changed. For this demo that's harmless. In a more complex system with expensive organisms or large worlds, you'd want the tick to only do real work if something changed — or separate "time has passed" as a distinct signal that only the organisms who care about it subscribe to.
find_organism is a linear scan by name. This runs every cycle, multiple times per cycle (in maintain_judge alone it runs five times). Right now with five organisms it doesn't matter. It's worth naming as a structural assumption that doesn't scale, even if it never needs to in practice.
The marquee organism skips HOLD_RESOURCE — it transitions from ARMED to SELECTING without claiming the pointer through the resource system, just transitioning state directly. This is probably intentional (marquee doesn't hold a world object, just the pointer), but it means its coordination path is structurally different from the drag organisms without that difference being architecturally explained. A future reader has to infer why.

The architecture's core insight — that interaction is a coordination problem, not just an event routing problem — is sound and underexplored in mainstream UI frameworks. The weaknesses are mostly about the gap between the conceptual generality of the model and the concreteness of the current implementation. The judge in particular is doing real architectural work but is implemented as if it's bespoke to these five organisms. That's the place I'd think hardest about if this were going to scale.

== On Required Disciplines ==
date: 2026-03-23

The organisms being task-focused is only possible because the judge exists. If organisms had to think about each other, you'd lose both priorities at once: organisms become entangled, and the system becomes rigid because every new organism has to know about all the existing ones. The judge is what makes organism-isolation sustainable as the system grows.
And the judge being hand-written is what makes it actually flexible rather than just theoretically flexible. A general rule engine sounds more flexible but isn't — it can only handle the cases its rules can express, and you end up fighting the rule language when something genuinely novel appears. A hand-written judge can say "in this specific situation, resize wins over drag, and here's why" in plain readable code. That's not a limitation, that's expressiveness.
The one tension worth naming: as the number of organisms grows, the judge grows too, and a hand-written judge can become hard to read. The mitigation isn't to make it less hand-written — it's to make sure the judge is reading clean declarations (well-structured HELDs) and speaking in a clear vocabulary of coordination primitives, so the policy logic stays legible even as it grows. The judge can be long without being complicated, if each case reads clearly.
So the architecture is sound. The work is in the discipline of keeping HELD a clean public interface, and keeping the judge's language expressive enough that its policy is readable to a future human who needs to extend it.
